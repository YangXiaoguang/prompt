> 您将获得：
> 使用特定行引用，对所有 24 条规则进行通过/失败检查。
> 每个发现问题的严重程度评级（严重/高/中/低）
> 针对每项违规行为，提供可直接实施的代码修复方案
> 修复每个问题的预计时间

<security_rules>
认证与会话管理：
01. 会话生命周期：JWT 会话有效期不得超过 7 天；必须实施刷新令牌轮换机制。
02. 禁止使用 AI 构建认证：必须使用 Clerk、Supabase 或 Auth0 等成熟服务，不得自行实现。
03. API 密钥必须通过 process.env 使用——严禁硬编码、记录到日志或提交到代码库。

安全 API 开发：
04. 机密信息必须至少每 90 天轮换一次（检查代码中是否有轮换策略）。
05. 所有 npm 包在安装前必须确认是否存在已知漏洞。
06. 软件包版本必须保持最新且安全（不得锁定过时版本）。
07. 每次构建后必须运行 npm audit fix（检查 CI/CD 流水线中是否配置）。
08. 所有输入必须使用参数化查询——严禁直接拼接原始 SQL 字符串。

API 与访问控制：
09. 必须从一开始就启用数据库的行级安全策略。
10. 生产环境代码中不得包含 console.log 语句。
11. CORS 必须限制访问，仅允许列入白名单的生产域名。
12. 所有重定向 URL 必须经过白名单验证。
13. 每个端点都必须实施身份认证和速率限制——无一例外。

数据与基础设施：
14. AI API 调用成本必须在代码和仪表板层面设置上限。
15. 必须通过 Cloudflare 或 Vercel edge 配置 DDoS 防护。
16. 存储访问权限必须仅限于已通过身份认证的用户自身。
17. 文件上传必须通过文件签名进行验证，而不能仅依赖文件扩展名。
18. 在处理支付数据前，必须验证 Webhook 签名的有效性。

其他关键规则：
19. 所有安全检查必须在服务端强制执行，而不能仅在 UI 侧实现。
20. 关键操作必须记录日志。
21. 账户删除必须永久清除相关数据。
22. 必须存在备份自动化机制以及经过测试的恢复流程。
23. 测试环境和生产环境必须完全隔离。
24. Webhook 绝不能触及测试环境中的真实系统。
</security_rules>
